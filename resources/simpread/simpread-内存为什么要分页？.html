内存为什么要分页？
littleNewton

首先，假设一个 Computer 的 memory 空间是

个字节。

如果你的这个 Computer 只跑一个程序，即只有一个进程，那么完全没必要分页。因为所有的字节都是这个程序所有。然而，这很不现实，目前所有的现代处理器都支持多任务。接下来想想多任务场景下，所有进程共享一个物理内存编址是否可行呢？

答案是不太可行。理由如下。

如果你把一段内存分给进程 A，然后把这段内存之后的另一段内存分给进程 B。那么在 A 执行完释放内存占用之后，由于其内存编址在 B 之前，所以 B 不能直接把那段内存拿过来用。这就像你写字一样，有人在本子的前半部分写完擦除了，你在后半部分写完了，这时不能直接从最后一页接着往前写，因为地址要连续。

后来有个非常聪明的工程师发明了虚拟内存编址方案！这可是划时代的东西！有了它，程序员直接和虚拟内存编址（虚空间）打交道，不再自己管理系统内存了。每个程序都有自己的一个虚拟内存空间，范围是

. 当然，一个程序一般不会真用这么多内存，一般几十兆就够了。

这里我们重点讲讲为什么需要用到内存地址。有些同学在学操作系统时对计算机是如何完成一个计算过程不甚明了。我这里简单一提。图灵机给我们证明了，很多过程可以用纸带模型表示出来，那么这个纸带就是内存。纸带上的一小段，就是一个 byte（姑且这么理解）。图灵机的读写头内部能存储少量的状态，机头存储的状态就是寄存器。寄存器根据上次存储的内存位置去找对应的内存字节，找到之后读出来或者写入东西。这个过程就是 RISC 里的 Load/Store. 所以从这个过程来看，只要你的内存空间是连续的，哪怕有人劫持了你看到的空间，你也是可以完成计算过程的。

在一个进程自信地认为拥有完整的内存编址空间时，就可以为所欲为了！但是操作系统实际上劫持了所有进程的访存。当一个进程要读写内存时，CPU 就会通过查询的方式，确定进程请求的虚地址的实际物理地址，然后给以反馈，而进程也可以自由地在虚空间里遨游。这个过程就是劫持。当然，这增加了 CPU 的设计复杂度，引进了一些额外的开销。

但是要保存虚地址到物理地址的映射是比较困难的。如果做字节级别的映射，就得保留

条映射记录，而且必须是每个进程搞一套映射，毕竟进程的虚空间是完整的 32 位空间。但这显然不切实际，你物理内存才多大啊！

聪明的工程师觉得既然字节级别的映射不行，我就做更大单元的映射，于是就产生了页！一个虚空间的页对应一个物理空间的页，一个最小页是 4KB，即

字节。（也有少数是最小 16KB。）那么映射关系就只有
条，即 1KK=1M 条，假设每条映射需要 4 字节，总共也就 4MB，这个代价可以接受。

所以内存分页是很聪明的做法！

说点后话，为了实现 CPU cache 与 TLB 翻译的同时性，聪明的工程师利用了物理页、虚页的同一性原理，发明了 VIPT（Virtual Index and Physical Tag）这种模式，可以一边用虚地址的 Index 部分查询 cache 的所有 set，一边做 TLB 翻译物理地址。等所有 cache set 的 index 位都读出来之后，再用 TLB 拿到的物理地址的 tag 与所有 set 的读出结果做 compare. 岂不美哉~

总结：虚拟内存，真是好发明。分页无非是减少映射代价而已。

的虚拟地址，在页大小至少为 4KB、单个映射用 4Byte 保存的情况下，需要用 4MB 空间保存映射关系（页表）。但是现在都是 64 位系统，
，也就是 4KKKKK=4KKG=4K TB 个映射关系，这个数有多大呢？是 4PB，相信你家的所有硬盘加起来都没这么大。

所以在 64 位系统上用一层页表已经顶不住了，仅存放映射关系就是一件要命的事情。后来，聪明的工程师发明了多级页表，问题就迎刃而解了。

碎碎念：多级页表的概念理解起来不难，但是准确理解却不是很容易。本科我学过一遍，研究生又学了一遍，真正写过 Verilog 才知道为什么要用多级页表。核心就在于 last level 的页表要在内存中全部存起来！而中间级到最初级的页表没必要都存起来。这个认识很重要！这关乎你对 CPU 存储管理的认识。为什么最高层的页表要在内存中全部存放呢？
  • 你考虑一下，假设你有一本天书，共十章。从高到低按章 - 节结构索引。
  • 这本书有两级目录，一级目录在书的最前面，用以标记了每一章的章目录所在页码。我们管一级目录叫做所有章目录的目录。确实有点绕~
  • 二级目录位置就不确定了，得通过查刚才说的一级目录的才能确定所在位置。而且一级目录的每一个映射都对应一个二级目录。二级目录里存放了每一节的位置。二级目录叫做所有节的目录。
  • 在程序运行过程中，某些节是用不到的，或者你写程序时根本就用不到那么多内存。比如，你第八章所有的节都没用到。这时候你的最高层目录（一级目录，所有章目录的目录）里只有一个对第八章子目录的一个空索引。而你自己真正需要的第一章、第二章，就存在一级目录里有真实索引了！
  • 但是，无论如何，一级目录都是必须保存的。因为一个含 malloc 的交互式程序所需要的内存是不确定的。如果你不保存所有的一级目录，那么等你真需要某些章时，你发现 TLB 里没有索引，去页表区也没有索引，系统就认为你这是个不合法的地址。保存所有一级目录，就相当于告诉别人，你别看我这个书只有区区一页目录，但是第一章、第二章、…… 第十章都是全的（哪怕是空映射），你来查目录可能会有 Page Fault 例外，但是这也属于正常范畴，等我给你分配一个物理页即可。但如果你的全十章有部分缺失，比如第八章在一级目录里是空的，那就说明虚空间是不全的！这哪能让程序员放心编译呢！
