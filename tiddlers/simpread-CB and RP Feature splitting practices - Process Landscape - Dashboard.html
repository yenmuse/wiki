Purpose of this description is to offer feature splitting and dependency linking guidance for New RF HW introduction features.

New RF development work is divided into two backlog features:

  • System level development in CB-feature, which gets feature ID allocation CB00XXXX from Focal Point
  • Platform RF program development in RP-feature, which gets feature ID allocation RP00YYYY from Focal Point.

CB-feature has RF DU FOT leader allocation, and it covers needed system module development, L3 integration and system testing.
Customers will see only public CB-feature, and it will be released to customers via system release program.

RP-feature work is led by RF program manager, and RP-feature covers all the RF DU development, testing and production aspects needed for the new RF.
RP-feature has no FOT leader, but the RF program manager can be considered FOT leader of RP-feature. Sub-FOTs are used to split the work into cross-functional sub-projects.

CB-feature testing is fully dependent on the HW and RFSW schedules planned in RP-feature.
However, RP-feature should consider in its planning the needs of the CB-feature, and target to support the most optimal system verification plan.
Thus dependency management is very critical. RP-feature enabler items for each verification step must be identified and dependency must be documented in the backlog.
Target FP schedules for each RP-enabler item come from the CB-feature.

All new RF HW introduction features should follow this same linking guidance, and not invent their own methods.

4.1. User stories split

A User Story is a small, concise statement of functionality or quality needed to deliver value to a specific stakeholder. It captures the need of a specific stakeholder and enable teams to define features of value to a stakeholder using short, simple documentation. They can serve as a base for “needs identification” and allow for the prioritizing, estimation and planning of the solution.

User Stories can be used:

  • to capture stakeholder needs and prioritize development
  • as a basis of estimation and planning solutions deliver,
  • as a basis for generation user acceptance tests
  • as a metric for measuring the delivery value,
  • as a unit of tracing related requirements,
  • as a basis for additional analysis
  • as a unit of project management and reporting.

4.1.4. Acceptance Criteria

A user story may be supported through the development of detailed acceptance criteria. AC define the boundaries of a user story and help the team to understand what the solution needs to provide to deliver value for stakeholder. AC might be supplemented with other analysis models.

Acceptance and evaluation criteria define measures of values attributes to be used for assessing and comparing solutions and alternative designs. Measurable and testable criteria allow for the objective and consistent assessment of solutions and designs.

AC describe the minimum set of requirements that must be met in order for a particular solution to be worth of implementation and may be used determinate if solution or solution component can meet requirements.

Elements:

  • Value attributes are the characteristics of a solution that or substantially include its value for stakeholders. They represent a meaningful and agreed-upon decomposition of the value proposition into its constituent parts, which can be described as qualities that the solution should either possess or avoid.
  • Assessment – we need to construct acceptance criteria in a measurable format.
  • Testability – AC should be expressed in testable form. This may require breaking requirements down into an atomic form so the test cases can be written to verify the solution against criteria. AC need to present in form of statements which can be verified as true or false. This can be done by User Acceptance Testing.
  • Measures – Evaluation criteria provide a way to determinate if features provide the value necessary to satisfy stakeholder needs. The definition of criteria allows the solution to be measured through various of methods such as benchmarking or expert judgement, evaluation criteria may involve designing tools and instructions for performing the assessment, as we as for recording and processing its results.

CB-feature must prepare Feature Test Strategy as defined in MN FOT WoW: MN FOT community - Feature Test Strategy (sharepoint.com).
Feature test strategy scope is the components and verification steps planned under CB-feature. 
Thus in case of new RF HW introduction, RF DU testing like Beamforming verification is not in scope of CB-feature FTS document.

Feature Test Strategy is not mandatory for RP-feature / RF product program, and typically not created.
However, it could introduce benefits, especially if reviewed together with complementing CB-feature test strategy.
